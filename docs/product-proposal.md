# Product Proposal

## 1. Обоснование идеи

При написании длинных произведений автор удерживает большое количество взаимосвязанных сущностей: персонажей, событий, причинно-следственных связей, правил мира, временных переходов и отношений между героями. По мере роста текста появляются типичные проблемы:
- несостыковки в фактах;
- нарушение таймлайна;
- поведение персонажей, противоречащее ранее заданному характеру;
- потеря логической связи между сценами.

Существующие инструменты для писателей в основном помогают хранить заметки, но не принимают решений о проверках, не управляют многошаговым анализом и не ведут агентную память истории.

`Story Consistency Agent` решает эту задачу как агентная система:
- агент получает цель анализа;
- оценивает текущее состояние story memory;
- сам выбирает инструменты для проверки персонажей, таймлайна, связанных событий и внешних фактов;
- интерпретирует результаты;
- обновляет рабочее состояние;
- формирует вывод или предлагает обновление памяти.

Это делает проект агентом, а не обычным RAG-чатом и не жёстким workflow.

## 2. Цель проекта и метрики

### Цель проекта

Создать PoC агентной системы, которая автоматически поддерживает память мира истории и помогает автору обнаруживать логические и сюжетные несостыковки в новых сценах.

### Продуктовые метрики

- доля обнаруженных реальных несостыковок;
- доля замечаний, которые пользователь оценивает как полезные;
- среднее время анализа новой сцены;
- доля сцен, для которых агент смог дать объяснимый результат.

### Агентские метрики

- `tool selection accuracy`;
- `precision` и `recall` для детекции конфликтов;
- доля корректно обновлённых записей story memory;
- среднее число reasoning/tool steps на одну сцену;
- доля случаев, где агент корректно сообщает о неопределённости.

### Технические метрики

- `p95 latency` анализа сцены до 10 секунд;
- среднее число `tool calls` на одну сцену;
- токеновая стоимость анализа;
- доля ошибок инструментов, обработанных без падения сценария.

## 3. Потенциальные сценарии использования

### Сценарий 1. Анализ новой сцены

Автор добавляет новый фрагмент текста. Агент извлекает персонажей и события, ищет связанные записи в памяти истории, проверяет сцену на конфликты и возвращает объяснение.

Edge case:
персонаж действует способом, который противоречит ранее зафиксированным чертам характера.

### Сценарий 2. Проверка таймлайна

Автор пишет, что событие произошло "на следующий день". Агент сравнивает новый фрагмент с временной линией и выявляет, что предыдущие события указывают на другой интервал времени.

Edge case:
во фрагменте используются неоднозначные временные указания, и агент должен явно сообщить о низкой уверенности.

### Сценарий 3. Диалог с агентом по сцене

Автор выделяет фрагмент и задаёт вопрос: почему сцена выглядит странно или как персонаж, скорее всего, повёл бы себя в этой ситуации. Агент использует story memory и результаты предыдущих извлечений, чтобы дать объяснимый ответ.

Edge case:
в памяти недостаточно данных, поэтому агент должен предложить гипотезу, а не выдать её как факт.

### Сценарий 4. Внешний research

Если сцена затрагивает внешнюю специализированную тему, агент может вызвать внешний инструмент проверки фактов и сопоставить результат с контекстом истории.

Edge case:
внешние источники противоречат друг другу, и агент должен зафиксировать конфликт, а не выбирать источник без пояснения.

## 4. Ограничения

### Технические ограничения

- `p95 latency` анализа сцены до 10 секунд;
- ограниченный размер входа: текст в пределах нескольких глав для PoC;
- ограничение количества `tool calls` на один анализ;
- ограничение на глубину agent loop для контроля времени и стоимости.

### Операционные ограничения

- ограниченный бюджет токенов на сессию и на анализ сцены;
- ограниченное число внешних запросов;
- использование только открытых или явно разрешённых источников;
- ручное подтверждение всех изменений story memory.

### Ограничения, связанные с риском

- внешние данные считаются недоверенными;
- write-действия отделены от read-инструментов;
- агент не должен автоматически менять память без user confirmation.

## 5. Архитектурный набросок

Система состоит из следующих модулей:
- `Text ingestion` — загрузка текста произведения и новых сцен;
- `Entity extraction` — извлечение персонажей, событий, фактов и связей;
- `Story memory` — структурированное хранилище мира истории;
- `Consistency agent` — управляющий агентный цикл, принимающий решения о шагах и инструментах;
- `Tools layer` — набор семантических read/write tools;
- `Author interface` — точка входа для автора, вопросов и подтверждений.

### Логические интерфейсы системы

Для следующего этапа зафиксировать базовые контракты:
- `ingest_story(text)` — загрузка и первичный разбор текста;
- `analyze_scene(scene_text)` — анализ новой сцены;
- `query_story_memory(query)` — поиск фактов о мире истории;
- `propose_memory_update(changes)` — предложение обновления памяти;
- `confirm_memory_update(update_id)` — подтверждение записи пользователем;
- `external_fact_check(query)` — внешний research при необходимости.

### Почему это агентная архитектура

Проект соответствует минимальной формуле агентного проекта:
- агент принимает решения о действиях, а не следует полностью жёсткому сценарию;
- использует несколько инструментов;
- выполняет многошаговую задачу;
- поддерживает state и memory;
- работает в условиях ошибок и неопределённости;
- ограничен по latency, бюджету и числу tool calls;
- допускает evaluation по измеримым метрикам;
- состоит из нескольких архитектурных модулей.

## 6. Потенциальный data flow

1. Автор загружает историю или новый фрагмент.
2. `Text ingestion` передаёт текст в агентный контур.
3. Агент определяет подзадачу: извлечение, проверка памяти, проверка таймлайна, поиск противоречий или внешний research.
4. Агент выбирает и вызывает один или несколько инструментов.
5. Инструменты возвращают структурированные результаты и интерпретируемые ошибки.
6. Агент сопоставляет результаты, при необходимости повторяет шаги или меняет стратегию.
7. Агент формирует вывод: конфликт, объяснение, уровень уверенности, рекомендации.
8. Если обнаружено новое устойчивое знание, агент предлагает обновление story memory.
9. Обновление записывается только после подтверждения автора.

### Ответственность LLM

LLM отвечает за:
- извлечение сущностей и фактов из текста;
- интерпретацию сцены;
- выбор следующего действия в рамках agent loop;
- объяснение конфликта или неопределённости.

### Ответственность deterministic code

Код системы отвечает за:
- хранение story memory и task state;
- контроль read/write boundaries;
- валидацию обновлений памяти;
- ограничение числа `tool calls`, глубины цикла и бюджета;
- логирование, метрики и трассировку действий;
- принудительное подтверждение write-операций пользователем.
