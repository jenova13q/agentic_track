# Evaluation Summary

## Цель оценки

Для PoC `Story Consistency Agent` оценка строится вокруг трёх вопросов:
- корректно ли система извлекает структуру истории из нового фрагмента;
- корректно ли агент использует память и контекст при проверке сцены;
- корректно ли работает `pending -> confirm / reject` контур без скрытых записей в подтверждённую память.

## Что считается успешным результатом

Сценарий считается успешным, если одновременно выполняются следующие условия:
- система возвращает корректный класс результата: `conflict`, `no_conflict` или `uncertain`;
- правильно определяется тип проблемы, если конфликт найден: `character`, `fact`, `timeline`, `object` или `mixed`;
- из текста извлекаются ключевые сущности и связи в ожидаемых категориях:
  - персонажи;
  - места;
  - предметы;
  - события;
  - факты;
  - связи;
- retrieval подтягивает только релевантную подтверждённую память и ближайший текстовый контекст;
- `pending` и `rejected` обновления не попадают в canonical story memory без явного подтверждения;
- отклонённый фрагмент не влияет на последующий анализ и не появляется в полном тексте истории.

## Покрытые сценарии

Автотесты и ручные сценарии покрывают следующие классы кейсов:

### 1. Извлечение структуры сцены
- выделение персонажей, мест, предметов, событий, фактов и связей;
- отделение описания сцены от события;
- разделение `character trait`, `location fact`, `object state` и `scene description`;
- нормализация имён и сопоставление с уже подтверждённой памятью.

### 2. Агентный анализ сцены
- корректный verdict для `no_conflict`;
- `uncertain` при нехватке контекста или неразрешённых ссылках;
- `character conflict` внутри одного фрагмента;
- `object conflict` при противоречии с уже подтверждённым состоянием объекта;
- использование retrieval и chunk windows как части рабочего контекста агента;
- передача в LLM-путь структурированного extraction payload, включая `locations`.

### 3. Memory governance
- `pending-first` поток для любого нового фрагмента;
- подтверждение обновления и перевод в canonical story memory;
- отклонение обновления без побочных эффектов на подтверждённую память;
- отсутствие утечки `pending` и `rejected` записей в retrieval;
- отсутствие утечки `pending` и `rejected` фрагментов в полный текст истории и локальные chunk windows.

### 4. API и системный контур
- ingest истории;
- анализ нового фрагмента;
- confirm / reject pending update;
- observability summary и traces;
- базовый web UI.

- работоспособность orchestrator и verdict pipeline;
- корректность API и pending-confirm-reject потока.

## Сильные стороны текущего PoC

- Архитектура действительно memory-centric, а не сводится к одному prompt.
- Агент работает поверх структурированной памяти, retrieval и tools.
- Write path отделён от read path.
- Подтверждение обновлений вынесено в отдельный контролируемый контур.
- Есть автоматическая регрессия на ключевые классы ошибок и на утечки состояния.

## Известные ограничения

Текущие ограничения в основном связаны не с архитектурой, а с качеством малой модели на extraction и verdict steps:

- модель иногда путает `event` и `fact`, особенно если событие подразумевает устойчивый результат;
- модель может смешивать `character trait` и общий факт о персонаже;
- сложные нелинейные временные переходы остаются чувствительным местом;
- при нескольких связанных утверждениях об объекте модель может хуже восстанавливать порядок состояний;
- вложенные воспоминания, ретроспекции и смена временных планов в пределах одного абзаца остаются наиболее рискованным классом сцен.

## Интерпретация результата

Для текущего Milestone проект показывает:
- устойчивый системный контур;
- воспроизводимый pending-first workflow;
- покрытие ключевых типов конфликтов;
- формализованную оценку extraction, retrieval, verdict и memory governance.

При этом оценка должна интерпретироваться как оценка PoC-агента для поддержки консистентности истории, а не как обещание полного литературного понимания произвольного текста.
